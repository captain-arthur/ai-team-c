**Role:** researcher

## 조사 목적

**더 단순하고 강력한 운영 확신 프레임워크**를 설계하기 위해, 다음을 이해한다:

1. 실제 SRE 팀이 모니터링 의사결정을 어떻게 구조화하는가?
2. 선행 지표와 후행 지표의 본질적 차이는 무엇인가?
3. 고장 전파를 어떻게 모델링할 수 있는가?
4. 조기 경보 시스템의 핵심 원리는 무엇인가?
5. 신호 계층은 어떻게 설계해야 하는가?

---

## 조사 질문

- Q1: 왜 기존 모니터링 프레임워크(USE, RED, Golden Signals)는 **운영 확신**을 주지 못하는가?
- Q2: **인과 체인(causal chain)**을 어떻게 정의하고 검증할 수 있는가?
- Q3: **최소 신호 집합(minimal signal set)**을 어떻게 도출할 수 있는가?
- Q4: 신호를 **행동**으로 연결하는 좋은 모델은 무엇인가?

## 조사 결과

### 1. 기존 프레임워크의 한계

#### USE Method (Utilization, Saturation, Errors)

| 요소 | 설명 |
|------|------|
| **Utilization** | 리소스가 얼마나 사용되고 있는가 (예: CPU 70%) |
| **Saturation** | 리소스가 얼마나 포화되었는가 (예: queue length) |
| **Errors** | 에러가 발생하고 있는가 |

**한계:**
- **리소스 중심**: 시스템 리소스(CPU, 메모리, 디스크, 네트워크)에 초점. 워크로드나 서비스 관점이 약함.
- **판단 규칙 부재**: "Utilization이 70%면 괜찮은가?"에 대한 답이 없음. 숫자는 있지만 **결론**이 없음.
- **인과관계 부재**: U, S, E가 서로 어떻게 연결되는지 설명하지 않음.

#### RED Method (Rate, Errors, Duration)

| 요소 | 설명 |
|------|------|
| **Rate** | 초당 요청 수 |
| **Errors** | 에러율 |
| **Duration** | 요청 처리 시간 |

**한계:**
- **서비스 중심**: 개별 서비스의 동작에 초점. 인프라 문제나 클러스터 전체 건강을 보기 어려움.
- **후행 지표 중심**: Rate, Errors, Duration은 이미 **발생한 후**의 결과. 예측이 아니라 진단.
- **클러스터 모니터링에 부적합**: Kubernetes 클러스터 운영 관점에서 "노드가 건강한가?", "스케줄링이 정상인가?"를 다루지 않음.

#### Google SRE Golden Signals

| 요소 | 설명 |
|------|------|
| **Latency** | 요청 처리 시간 |
| **Traffic** | 요청량 |
| **Errors** | 에러율 |
| **Saturation** | 포화도 |

**한계:**
- RED + Saturation. 여전히 **서비스 레벨** 초점.
- **판단 프레임워크 아님**: "이 네 가지를 보라"고 하지만, "이것들이 어떤 상태면 안전한가?"를 말하지 않음.

**핵심 발견:**

- 기존 프레임워크는 **"무엇을 볼 것인가"**에 대한 답은 주지만, **"보고 나서 어떻게 판단할 것인가"**에 대한 답을 주지 않는다.
- **운영 확신**을 위해서는 metric 목록이 아니라 **판단 규칙**이 필요하다.
- 기존 프레임워크를 복사하는 것이 아니라, **Kubernetes 클러스터 운영**에 맞는 새로운 모델이 필요하다.

### 2. 선행 지표 vs 후행 지표의 본질

#### 정의

| 유형 | 정의 | 특성 |
|------|------|------|
| **선행 지표 (Leading)** | 결과가 발생하기 **전에** 변하는 신호 | 예측 가능, 사전 대응 가능 |
| **후행 지표 (Lagging)** | 결과가 발생한 **후에** 변하는 신호 | 진단에 유용, 사후 확인 |

#### 핵심 통찰: 원인-결과 체인에서의 위치

선행/후행은 **절대적 속성이 아니라 관점의 문제**다.

```
[원인 A] → [중간 B] → [결과 C]
```

- B 관점에서: A는 선행, C는 후행
- C 관점에서: A와 B 모두 선행

따라서 **"무엇에 대한 선행인가?"**를 명시해야 한다.

#### Kubernetes 클러스터에서의 예시

```
[노드 CPU 90%] → [스케줄링 지연] → [Pending pods 증가] → [서비스 응답 지연]
```

| 신호 | 서비스 영향에 대해 | 특성 |
|------|---------------------|------|
| 노드 CPU 90% | 선행 | 서비스 영향 전에 감지 가능 |
| 스케줄링 지연 | 선행 (더 가까움) | 영향 직전 단계 |
| Pending pods | 후행 (또는 동시) | 이미 스케줄 실패 발생 |
| 서비스 응답 지연 | 후행 | 이미 영향 발생 |

**핵심 발견:**

- 선행/후행은 **체인에서의 위치**로 결정된다.
- 선행 지표를 찾으려면 먼저 **인과 체인**을 정의해야 한다.
- 모니터링 설계는 "어떤 metric을 볼까?"가 아니라 **"인과 체인이 무엇인가?"**에서 시작해야 한다.

### 3. 고장 전파 모델 (Failure Propagation Model)

#### 핵심 개념: 단계적 전파

시스템 문제는 **한 번에** 발생하지 않는다. **단계적으로 전파**된다.

```
[Pressure Stage] → [Symptom Stage] → [Failure Stage] → [Impact Stage]
```

| 단계 | 설명 | 예시 |
|------|------|------|
| **Pressure** | 시스템에 스트레스가 가해짐 | 부하 증가, 리소스 사용률 상승 |
| **Symptom** | 스트레스의 첫 번째 증상 | 지연 증가, 큐 쌓임, 경고 |
| **Failure** | 실제 실패 발생 | 에러, 타임아웃, 재시작, NotReady |
| **Impact** | 사용자/비즈니스 영향 | 서비스 다운, 매출 손실 |

#### Kubernetes 클러스터에 적용

```
[Resource Pressure] → [Scheduling Strain] → [Workload Instability] → [Service Impact]
    리소스 압박            스케줄링 어려움          워크로드 불안정          서비스 영향
```

| 단계 | Kubernetes 현상 | 대표 신호 |
|------|-----------------|-----------|
| **Resource Pressure** | 노드 CPU/메모리/디스크 사용률 상승 | Node CPU > 80%, Memory > 80%, Disk < 10% |
| **Scheduling Strain** | 스케줄러가 파드를 배치하기 어려움 | Pending pods 증가, Scheduler latency |
| **Workload Instability** | 파드가 불안정해짐 | Restarts 증가, OOM kills, Evictions |
| **Service Impact** | 서비스가 영향받음 | Endpoints 감소, 응답 지연, 5xx |

**핵심 발견:**

- 고장은 **4단계 체인**으로 모델링할 수 있다: Pressure → Strain → Instability → Impact
- 각 단계에는 **대표 신호**가 있다.
- **앞 단계를 보면 예측**, **뒷 단계를 보면 진단**이 가능하다.

### 4. 조기 경보 시스템의 원리

#### 핵심 개념: 시간적 여유 (Lead Time)

조기 경보의 가치는 **행동할 시간**을 주는 것이다.

```
[경보 발생] ─────── Lead Time ───────> [영향 발생]
                    ↑
              이 시간 동안 대응 가능
```

#### 좋은 조기 경보의 조건

| 조건 | 설명 |
|------|------|
| **충분한 Lead Time** | 대응할 시간이 있어야 함 |
| **높은 정밀도** | 거짓 경보(false positive)가 적어야 함 |
| **명확한 행동** | 경보를 보고 **무엇을 해야 하는지** 알아야 함 |

#### Kubernetes에서의 적용

| 신호 | Lead Time | 정밀도 | 행동 |
|------|-----------|--------|------|
| Node CPU > 80% | 높음 (분~시간) | 중간 | 워크로드 분산, 노드 추가 |
| Pending pods 증가 | 중간 (분) | 높음 | 리소스 확장, 우선순위 조정 |
| Restarts 폭증 | 낮음 (이미 발생) | 높음 | 원인 조사, 롤백 |
| Endpoints 비어 있음 | 매우 낮음 | 매우 높음 | 즉시 대응 |

**핵심 발견:**

- **Lead Time이 긴 신호**일수록 조기 경보로서 가치가 높다.
- 신호를 **Lead Time 순서**로 배열하면 자연스러운 우선순위가 생긴다.
- 조기 경보는 **행동과 연결**되어야 의미가 있다.

### 5. 신호 계층 설계 원칙

#### 핵심 개념: 계층적 추상화

복잡한 시스템을 이해하려면 **계층적 추상화**가 필요하다.

```
[상위 계층] ← 요약/결론
    ↓
[중간 계층] ← 원인/맥락
    ↓
[하위 계층] ← 상세/조사
```

#### 좋은 신호 계층의 조건

| 조건 | 설명 |
|------|------|
| **상위는 결론** | 상위 계층만 보면 "안전한가?"에 답할 수 있어야 함 |
| **하위는 원인** | 하위로 갈수록 "왜?"에 답할 수 있어야 함 |
| **계층 간 연결** | 상위 이상 → 해당 하위로 드릴다운 가능해야 함 |

#### ai-team-lab의 Block 모델 재검토

| Block | 역할 | 문제점 |
|-------|------|--------|
| Block 1 (Operational Confidence) | 지금 안전한가? | 여러 신호를 AND로 묶음. 왜 이 신호들인지 직관적이지 않음 |
| Block 2 (Early Risk) | 곧 위험한가? | Block 1과의 관계 불명확. 둘 다 "현재 상태"를 본다는 점에서 중복 |
| Block 3 (Investigation) | 어디를 볼까? | 드릴다운이지만, Block 1/2와의 매핑이 명확하지 않음 |

#### 개선 방향: 체인 기반 계층

인과 체인의 **위치**를 기준으로 계층을 나누면 더 직관적이다:

```
[Early Stage]  →  [Middle Stage]  →  [Late Stage]
   예측용             진행 중              진단용
```

**핵심 발견:**

- 계층은 **인과 체인의 단계**와 일치해야 자연스럽다.
- "안전한가?"는 **Late Stage가 정상인가?**로 답할 수 있다.
- "내일 안전한가?"는 **Early Stage에 경고가 있는가?**로 답할 수 있다.
- "무엇을 해야 하나?"는 **어느 Stage에 이상이 있는가?**로 결정된다.

### 발견된 핵심 원칙

| # | 원칙 | 설명 |
|---|------|------|
| 1 | **체인 우선** | metric 목록이 아니라 **인과 체인**을 먼저 정의한다 |

| 4 | **행동 연결** | 모든 이상 신호는 **구체적 행동**으로 연결된다 |

## 결론

### 핵심 발견 요약

1. **기존 프레임워크(USE, RED)는 판단 규칙이 없다** — metric 목록만 제공하고 "그래서 괜찮은가?"에 답하지 않음.

2. **선행/후행은 인과 체인에서의 위치다** — 체인을 정의해야 선행/후행을 구분할 수 있음.

3. **고장은 4단계로 전파된다** — Pressure → Strain → Instability → Impact.

4. **조기 경보의 가치는 Lead Time이다** — 체인의 앞쪽일수록 대응 시간이 길다.

5. **계층은 체인 단계와 일치해야 한다** — 자연스러운 드릴다운과 의사결정이 가능해짐.

### 권장 사항

1. **4단계 고장 전파 체인**을 프레임워크의 중심 모델로 채택
2. 각 단계에 **대표 신호 1~3개**만 선정
3. 각 단계를 **하나의 운영 질문**에 매핑
4. 모든 이상 상태에 **명시적 행동**을 연결
5. Block1/Block2/Block3 대신 **체인 기반 계층**으로 재설계
