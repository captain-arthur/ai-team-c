# Problem Definition

**Role:** manager
**Date:** 2026-03-17

---

## 목표

Kubernetes 클러스터의 **운영 확신(Operational Confidence)**을 제공하는 프레임워크를 설계한다.

프레임워크는 다음 세 가지 질문에 **명확하고 즉각적으로** 답할 수 있어야 한다:

| # | 질문 | 운영적 의미 |
|---|------|-------------|
| 1 | **지금 안전한가?** | 현재 상태 판단 |
| 2 | **내일도 안전할 것인가?** | 미래 리스크 예측 |
| 3 | **무엇을 해야 하는가?** | 행동 결정 |

---

## 배경

### 가시성은 있지만 확신이 없다

Prometheus와 Grafana로 수많은 metric을 수집하고 대시보드를 운영해도, 운영자는 여전히 **"그래서 괜찮은 건가?"**를 묻는다.

- metric은 넘쳐나지만 **무엇이 중요한지** 모른다
- 대시보드 패널은 많지만 **한눈에 결론**을 내릴 수 없다
- 경고는 울리지만 **어떤 행동**을 해야 하는지 불분명하다

### ai-team-lab의 시도

이전 작업(ai-team-lab)에서 이 문제를 해결하려 시도했다:

- **Operational Confidence Theory**: 5가지 운영 안전 조건(C1-C5) 정의
- **Cluster Health Monitoring Model**: 3계층 구조(Summary / Trend / Top Offenders)
- **Central Kubernetes Operational Dashboard Design**: Block1/Block2/Block3 모델

이 작업들은 올바른 방향이지만, **아직 완성되지 않았다**.

---

## 현재 상황: 무엇이 불충분한가

### 1. 복잡도 문제

현재 설계는 **설명하기에 너무 복잡**하다:

- 5가지 안전 조건(C1-C5)
- 4가지 고장 전파 경로
- 3개의 블록(Block1/Block2/Block3)
- 다양한 신호 역할(safety condition, early warning, instability, damage, investigation)

다른 사람에게 "이 프레임워크가 어떻게 동작하나요?"를 물으면, 5분 안에 **명쾌하게** 설명하기 어렵다.

### 2. 인과관계 불명확

현재 설계는 **신호 간의 인과관계**가 명확하지 않다:

```
A가 발생하면 → B가 발생하고 → C가 발생하고 → D(서비스 영향)에 이른다
```

이런 **명시적인 인과 체인**이 정의되어 있지 않다. 고장 전파 경로가 서술되어 있지만:

- 각 경로가 **독립적으로** 기술되어 있음
- 경로 간의 **교차점**이 불명확
- **어떤 신호가 어느 단계에 해당하는지** 직관적이지 않음

### 3. 의사결정 연결 부족

현재 설계는 **"무엇을 해야 하는가?"**에 약하다:

- Block1이 비정상이면 "조사 필요"라고 하지만, **구체적으로 어떤 조사**를 해야 하는지 불명확
- Block2 경고가 있으면 "사전 대응"이라고 하지만, **어떤 대응**인지 불명확
- 신호를 보고 → 행동을 결정하는 **명확한 의사결정 트리**가 없음

### 4. 설득력 부족

현재 설계는 **리더십에 보고하기에 약하다**:

- "왜 이 신호들인가?"에 대한 답이 **기술적으로는** 있지만, **비즈니스 언어로** 설명하기 어려움
- "이 대시보드를 보면 무엇을 알 수 있나요?"에 대한 **한 문장 답변**이 없음
- 투자 대비 효과(ROI)를 설명하기 어려움

## 원하는 상태

### 단순한 모델

프레임워크는 **핵심 아이디어 하나**로 설명할 수 있어야 한다:

> "클러스터 안전은 **하나의 체인**으로 이해할 수 있습니다. 체인의 앞쪽을 보면 미래를 예측하고, 뒤쪽을 보면 현재를 진단합니다."

### 명확한 인과관계

신호 간의 관계가 **하나의 체인 또는 트리**로 시각화 가능해야 한다:

```
[Pressure] → [Symptoms] → [Instability] → [Impact]
```

각 신호가 이 체인의 **어느 위치**에 있는지 명확해야 한다.

### 행동 중심

모든 신호는 **행동**으로 연결되어야 한다:

| 신호 상태 | 결론 | 행동 |
|-----------|------|------|
| 체인 앞쪽 경고 | 곧 문제 발생 가능 | 용량 확장 / 정리 |
| 체인 중간 경고 | 문제 진행 중 | 원인 조사 |
| 체인 뒤쪽 경고 | 이미 영향 발생 | 즉시 대응 |

### 설명 가능

리더십에게 5분 안에 다음을 설명할 수 있어야 한다:

1. 이 프레임워크가 **무엇을 하는지**
2. 왜 **이것만으로 충분한지**
3. 어떻게 **운영 안전을 보장하는지**

## 성공 기준

이 프로젝트가 성공하려면:

- [ ] **단순성**: 프레임워크를 5분 안에 설명할 수 있다
- [ ] **인과관계**: 신호 간의 관계가 하나의 모델로 시각화된다
- [ ] **행동 연결**: 모든 신호가 구체적인 행동으로 매핑된다
- [ ] **최소성**: 필수 신호가 10개 이하로 정의된다
- [ ] **설득력**: 리더십에게 설명할 수 있는 executive summary가 존재한다
- [ ] **실용성**: 실제 운영에서 즉시 사용 가능하다

---

## 범위

### 포함 (In Scope)

- 신호 계층 구조(signal hierarchy) 설계
- 고장 전파 체인(failure propagation chain) 모델
- 최소 신호 집합(minimal signal set) 정의
- 신호-행동 매핑(signal-to-action mapping)
- 프레임워크 문서 및 executive summary

### 제외 (Out of Scope)

- Grafana 대시보드 JSON 구현
- Alert rule 상세 정의
- Runbook 작성
- 특정 환경(EKS, GKE 등)에 대한 커스터마이징

---

## 제약 조건

- **기존 작업 존중**: ai-team-lab의 올바른 아이디어는 유지하고 정제한다
- **실용성 우선**: 이론적 완벽함보다 실제 사용 가능성을 우선한다
- **기술 중립**: 특정 도구(Prometheus, Grafana)에 종속되지 않는 개념 모델을 만든다

---

## 가정

- 클러스터 모니터링에 Prometheus 또는 유사한 metric 시스템이 사용된다
- kube-state-metrics, node_exporter 등 기본적인 metric이 수집된다
- Managed Kubernetes(EKS, GKE, AKS)에서는 control plane metric이 없을 수 있다
