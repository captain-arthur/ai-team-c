# review-checklist

## 목적

작업 결과물의 품질을 체계적으로 검토합니다.

## 사용 시점

- 설계 완료 후
- 구현 완료 후
- 문서 완료 후
- 프로젝트 마무리 전

## 입력

- 검토 대상 (설계, 코드, 문서 등)
- 원래 요구사항 (`01-problem.md`)

## 절차

### 1. 검토 유형 선택

| 유형 | 대상 | 주요 확인 사항 |
|------|------|----------------|
| design-review | 설계 문서 | 요구사항 충족, 트레이드오프 |
| code-review | 코드 | 정확성, 가독성, 보안 |
| doc-review | 문서 | 명확성, 완전성 |

### 2. 설계 검토 (design-review)

```markdown
## 요구사항 충족
- [ ] 모든 기능 요구사항 반영
- [ ] 비기능 요구사항 고려
- [ ] 제약 조건 준수

## 설계 품질
- [ ] 단순성: 불필요한 복잡도 없음
- [ ] 확장성: 향후 변경 용이
- [ ] 일관성: 기존 시스템과 조화

## 리스크
- [ ] 주요 리스크 식별됨
- [ ] 대응 방안 존재

## 결정 사항
- [ ] 트레이드오프 명시됨
- [ ] 결정 이유 기록됨
```

### 3. 코드 검토 (code-review)

```markdown
## 정확성
- [ ] 요구사항대로 동작
- [ ] 엣지 케이스 처리
- [ ] 에러 처리 적절

## 가독성
- [ ] 명확한 명명
- [ ] 적절한 구조
- [ ] 필요한 주석만 존재

## 보안
- [ ] 입력 검증
- [ ] 인증/인가 확인
- [ ] 민감 정보 노출 없음

## 유지보수성
- [ ] 테스트 가능한 구조
- [ ] 중복 최소화
- [ ] 의존성 명확
```

### 4. 문서 검토 (doc-review)

```markdown
## 완전성
- [ ] 필요한 정보 포함
- [ ] 누락된 섹션 없음

## 명확성
- [ ] 이해하기 쉬움
- [ ] 모호한 표현 없음

## 정확성
- [ ] 사실과 일치
- [ ] 최신 상태
```

### 5. 검토 결과 기록

`05-review.md`에 기록:

```markdown
# Review: {대상}

## 검토 일시
YYYY-MM-DD

## 검토 유형
{design/code/doc}-review

## 결과 요약
- 통과 항목: N개
- 개선 필요: N개

## 개선 필요 사항
1. {항목}: {설명}
2. {항목}: {설명}

## 권장 사항
- ...

## 결론
{승인 / 수정 후 재검토 / 재작업 필요}
```

## 출력

- 검토 결과 문서
- 개선 사항 목록
- 승인/반려 결정

## 출력 위치

```
projects/{project}/05-review.md
```
