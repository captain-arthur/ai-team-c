# architecture-design

## 목적

문제에 대한 기술적 해결 방안을 설계하고 문서화합니다.

## 사용 시점

- 새로운 시스템/기능 설계
- 기존 시스템 개선 설계
- 기술 선택 및 의사결정
- 복잡한 문제의 구조적 해결

## 입력

- 문제 정의 (`01-problem.md`)
- 조사 결과 (`02-research.md`, 있는 경우)
- 제약 조건 및 요구사항

## 절차

### 1. 요구사항 정리

```markdown
## 기능 요구사항
- FR1: ...
- FR2: ...

## 비기능 요구사항
- NFR1: 성능 - ...
- NFR2: 확장성 - ...

## 제약 조건
- C1: ...
```

### 2. 옵션 도출

가능한 접근 방식 2-3개 식별:

```markdown
## Option A: {이름}
- 설명: ...
- 장점: ...
- 단점: ...

## Option B: {이름}
- 설명: ...
- 장점: ...
- 단점: ...
```

### 3. 트레이드오프 분석

| 기준 | Option A | Option B |
|------|----------|----------|
| 복잡도 | | |
| 유지보수성 | | |
| 성능 | | |
| 확장성 | | |

### 4. 설계 결정

선택한 옵션과 이유:

```markdown
## 결정
Option A 선택

## 이유
- 이유 1
- 이유 2

## 트레이드오프 수용
- 수용하는 단점과 그 이유
```

### 5. 상세 설계

```markdown
## 구조
- 컴포넌트 구성
- 데이터 흐름
- 인터페이스

## 구현 계획
1. 단계 1: ...
2. 단계 2: ...

## 주의사항
- 구현 시 고려할 점
```

### 6. 문서화

`03-architecture.md`에 기록

## 출력

- 설계 문서
- 구현 계획
- 의사결정 기록

## 출력 위치

```
projects/{project}/03-architecture.md
```

- 의사결정 교훈 → `knowledge/lessons-learned/`
